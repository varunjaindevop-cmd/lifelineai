# SAGE Accident Detection System - Upgrade Plan

## Executive Summary

This plan addresses three critical issues in the SAGE accident detection system:
1. **False alarms** in isolated road and traffic modes
2. **Browser unresponsiveness** (20-30s freezes) due to main-thread computation
3. **Outdated AI model** (COCO-SSD) that needs replacement with YOLOv8n

Plus: a Python training pipeline for custom Indian road accident detection.

---

## PHASE 1: YOLOv8n ONNX Integration

### Goal
Replace COCO-SSD (MobileNet v2) with YOLOv8n running via ONNX Runtime Web for better accuracy and speed.

### Files to Create

**`lib/detection/yolo-engine.ts`** (NEW - ~200 lines)
- YOLOv8n ONNX inference wrapper using `onnxruntime-web`
- Pre-processing: letterbox resize to 640x640, BGR/RGB conversion, normalization to [0,1]
- Post-processing: NMS (Non-Maximum Suppression), confidence threshold (0.25), class filtering
- COCO class mapping: person=0, bicycle=1, car=2, motorcycle=3, bus=5, truck=7
- Export classes to our unified format: car/truck/bus -> "car", motorcycle/bicycle -> "motorcycle", person -> "person"
- Support both standard COCO YOLOv8n AND custom-trained model loading
- Model loading: `ort.InferenceSession.create('/models/yolov8n.onnx')` with WebGL backend preference, WASM fallback

**`lib/detection/yolo-postprocess.ts`** (NEW - ~80 lines)
- `letterbox(img, targetSize=640)` - resize with padding
- `nms(boxes, scores, iouThreshold=0.45)` - Non-Maximum Suppression
- `processOutput(tensor, origWidth, origHeight)` - convert YOLO output tensor to detection array
- `deduplicateDetections(detections, mergeDistance=0.5)` - merge overlapping same-class detections

**`lib/detection/detection-worker.ts`** (NEW - ~150 lines)
- Web Worker that loads YOLOv8n model once
- Receives ImageBitmap via `postMessage` with Transferable
- Runs ONNX inference on worker thread (zero main-thread blocking)
- Returns detection results as structured data
- Handles model switching (standard vs custom)

### Files to Modify

- Remove `@tensorflow/tfjs` and `@tensorflow-models/coco-ssd` (Phase 1b, after verification)

**`app/(dashboard)/admin/videos/page.tsx`**
- Replace COCO-SSD `detect()` call with YOLOv8n worker messages
- Lines 65-86: Replace TensorFlow model loading with YOLO worker initialization

- Lines 26-30: Update COCO_MAP to match YOLOv8n class IDs
- Lazy-load: Show "Start Analysis" button immediately, load model on first click

### Detection Quality Improvements

**YOLOv8n advantages over COCO-SSD:**
- ~40% better mAP (mean Average Precision)
- Better small object detection (critical for distant vehicles)
- Better bounding box accuracy (tighter boxes = better overlap/IoU calculations)
- Faster inference (~30ms vs ~80ms per frame)

## PHASE 2: Web Worker + OffscreenCanvas Architecture

### Goal
Move ALL heavy computation off the main thread so the browser stays responsive during analysis.

### Architecture

```
Main Thread (UI only)          Web Worker (all CV)
========================       =====================
- Video element                - YOLOv8n inference
- Canvas overlay drawing       - Kalman tracking
- React state/UI               - TTC collision detection
- User interactions            - Speed estimation
                               - Preprocessing (CLAHE/denoise)
                               - Change detection grid
                               - Frame buffer management
```

### Files to Create

**`lib/worker/detection-worker.ts`** (NEW - ~400 lines)
- Self-contained detection pipeline that runs entirely in a Web Worker
- Message protocol:
  - `INIT` - Load model, set environment config
  - `FRAME` - Receive ImageBitmap, run detection, return results
  - `SET_MODE` - Switch isolated/traffic/marketplace
  - `STOP` - Stop analysis, clear state
- Contains: YOLO inference, Kalman tracker, TTC engine, speed estimator, change detection
- Uses `OffscreenCanvas` for frame capture (no `getImageData` on main thread)
- Returns only lightweight results: entities, evidence, change grid, state

**`lib/worker/message-types.ts`** (NEW - ~60 lines)
- TypeScript interfaces for worker message protocol
- `WorkerInput`: { type: 'INIT' | 'FRAME' | 'SET_MODE' | 'STOP', ... }
- `WorkerOutput`: { type: 'RESULTS', entities, evidence, changeGrid, state, fps }
- Type-safe message handling

**`lib/worker/frame-scheduler.ts`** (NEW - ~40 lines)
- `ThrottledFrameSender` class
- Captures video frame as ImageBitmap via `createImageBitmap(video)`
- Transfers ImageBitmap to worker (zero-copy via Transferable)
- Adaptive frame rate: skips frames under load (target 3-5 FPS detection)
- Uses `requestAnimationFrame` with frame skip counter

### Files to Modify

**`app/(dashboard)/admin/videos/page.tsx`** (Major refactor)
- **Remove**: All detection logic (~200 lines) from this component
- **Keep**: UI rendering, canvas overlay drawing, incident display, controls
- **Add**: Worker message handling, result state management
- **Simplify**: `startAnalysis` creates worker, sends frames; `stopAnalysis` terminates worker
- The `drawFrame` function stays on main thread (it needs the visible canvas)
- Detection results come back as messages, React renders them

**Before (~660 lines with detection logic):**
```
startAnalysis() {
  // 100+ lines of detection setup
  const loop = () => {
    scheduleDetection(video);        // TF inference on main thread
    tracker.update(detections);       // Tracking on main thread
    detectAccidents(entities);        // Collision on main thread
    drawFrame(entities, evidence);    // Drawing on main thread
    // All in one RAF loop = main thread blocked
  }
}
```

**After (~300 lines, detection moved to worker):**
```
startAnalysis() {
  worker = new Worker('/detection-worker.js');
  worker.postMessage({ type: 'INIT', mode: envMode });
  
  const loop = () => {
    createImageBitmap(video).then(bitmap => {
      worker.postMessage({ type: 'FRAME', bitmap }, [bitmap]);
    });
  };
  
  worker.onmessage = (e) => {
    const { entities, evidence, state } = e.data;
    drawFrame(entities, evidence);  // Only drawing on main thread
    updateUI(state);
  };
}
```

### Performance Gains

| Operation | Before (Main Thread) | After (Worker) |
|-----------|---------------------|----------------|
| YOLO inference | ~30ms blocking | ~30ms non-blocking |
| Kalman update | ~5ms blocking | ~5ms non-blocking |
| TTC detection | ~10ms blocking | ~10ms non-blocking |
| Preprocessing | ~15ms blocking | ~15ms non-blocking |
| Frame capture | `getImageData` GPU stall | `createImageBitmap` async |
| **Total per frame** | **~60ms main thread** | **~0ms main thread** |
| **UI responsiveness** | **Frozen 60ms/frame** | **Always responsive** |

### Additional Performance Fixes

**Frame Buffer Memory Optimization**
- Current: 90MB in-memory buffer (15 sec at 5 FPS)
- Fix: Use `Blob`-based buffer (compressed WebM segments) instead of raw ImageData
- Or: Reduce to 10-second buffer, lower resolution (320x240 for buffer only)

**Lazy Model Loading**
- Don't load model on page mount
- Show "Start Analysis" immediately
- Load model on first click, show spinner during load
- Cache loaded model in worker (shared across analyses)

**Fix Memory Leaks**
- `cameras/[id]/page.tsx` line 216: Clear interval in `stopAnalysis`
- Revoke object URLs with `URL.revokeObjectURL()`
- Clear frame buffer on analysis stop

## PHASE 3: False Alarm Fixes

### Goal
Eliminate false positives while keeping detection fast enough for real accidents.

### 3A. Isolated Road False Alarm Fix

**Root Cause**: `scoreIsolated()` in `ttc-engine.ts` line 60-94 triggers when a car passes in front of people because:
- Car passes close to person = overlap detected (+0.5)
- Car may decelerate slightly = deceleration detected (+0.4)
- Passing penalty only checks same-direction fast objects

**Fix: Enhanced Isolated Road Scoring**

**`lib/detection/ttc-engine.ts`** - Modify `scoreIsolated()`:

```
New signals to add:
1. TRAJECTORY INTERSECTION CHECK:
   - Predict where each object will be in N frames using Kalman prediction
   - Do predicted paths actually cross? If not, it's just passing.
   - Score: +0.3 for actual path intersection

2. DIRECTION CONVERGENCE:
   - Not just same direction (current passing penalty)
   - Check: are objects heading TOWARD each other or PAST each other?
   - Use velocity dot product: negative = toward each other (danger)
   - Positive = same direction (safe passing)
   - Score: +0.4 for convergence, -0.5 for passing

3. RELATIVE VELOCITY CHECK:
   - In a real collision, relative speed drops to near-zero at contact
   - In a pass, relative speed stays constant
   - Monitor: dist(a,b) over time. Decreasing = approaching, stable = passing
   - Score: +0.3 for rapidly decreasing distance

4. POST-CONTACT BEHAVIOR:
   - After real collision: both objects decelerate/stop
   - After passing: both continue at similar speed
   - Check: does either object stop or change direction after overlap?
   - Score: +0.2 for post-contact deceleration
```

**Updated scoring formula:**
```
score = trajectory_interception (+0.3)
      + direction_convergence (+0.4)
      + deceleration (+0.4 each, max 0.8)  
      + touching (+0.2)
      + relative_velocity_drop (+0.3)
      + post_contact_behavior (+0.2)
      - passing_penalty (-0.8 if clearly passing)

Threshold: 0.6 (up from 0.5)
Required: direction_convergence MUST be positive (toward each other)
```

**Key anti-false-positive guard:**
```
if (direction === "passing" || direction === "parallel") {
  return null;  // No alert regardless of overlap
}
```

### 3B. Traffic Mode False Alarm Fix

**Root Cause**: `detectTrafficAlerts()` in `ttc-engine.ts` line 101-136 triggers when a car brakes normally near another car because it only checks "was fast, now stopped, near another object."

**Fix: Energy Transfer Detection**

**`lib/detection/ttc-engine.ts`** - Enhance `detectTrafficAlerts()`:

```
New checks to add:
1. MUTUAL DECELERATION:
   - Not just ONE car stopping - did BOTH cars show velocity change?
   - Real collision: energy transfers, both affected
   - Normal braking: only one car slows, other continues
   - Require: nearby object also shows speed change within 2 frames

2. SCENE CHANGE CORRELATION:
   - Real collision causes visible scene disruption (camera shake, debris)
   - Use change detection grid: real collision shows concentrated energy spike
   - Normal braking: no scene change
   - Require: scene change spike > 0.3 in cells near the objects

3. APPROACHING BEFORE STOP:
   - Must have been approaching the other object (not just stopped nearby)
   - Check: distance was decreasing before the stop
   - If distance was stable = car just stopped in traffic, not a collision

4. MOMENTUM ANALYSIS:
   - Kinetic energy before: 0.5 * mass * v^2
   - If energy was high and now zero = impact
   - If energy was moderate and now zero = normal stop
   - Scale by vehicle class (car = 1.0, motorcycle = 0.3, truck = 1.5)
```

**Updated traffic detection:**
```
function detectTrafficAlerts(entities) {
  for (const entity of entities) {
    // Existing: dramatic stop check
    if (maxRecent <= 4 || recentSpeed >= 0.3) continue;
    
    // NEW: find nearby object
    const nearby = findNearby(entity);
    if (!nearby) continue;
    
    // NEW: mutual deceleration check
    const nearbyDecelerated = hasDecelerated(nearby);
    if (!nearbyDecelerated) continue;  // KEY: other object must also be affected
    
    // NEW: approach check
    const wasApproaching = distanceDecreasing(entity, nearby);
    if (!wasApproaching) continue;  // Must have been getting closer
    
    // NEW: scene change check
    if (sceneChangeNear(entity) < 0.2) continue;  // Must have visible impact
    
    // Only now trigger alert
    evidence.push({...});
  }
}
```

### 3C. Enhanced Granular Detection

**`lib/detection/ttc-engine.ts`** - Improve existing detectors:

**Person Fall (line 140-176):**
- Current: aspect ratio change + Y drop + was moving
- Add: **sustained lying detection** - person stays horizontal for 10+ frames = confirmed fallen
- Add: **interaction with vehicle** - was there a vehicle nearby when aspect ratio changed?
- Add: **velocity discontinuity** - person was moving, suddenly stopped + changed aspect ratio = hit by vehicle

**Bike Off-Track (line 180-215):**

- Add: **rider separation** - if person detection appears near fallen motorcycle

### 3D. State Machine Improvements

**`lib/detection/ttc-engine.ts`** - State machine in `detectAccidents()`:

```
Current: consecutive >= 2 frames -> alert
New: tiered response system:

Frame 1-2: "suspecting" - evidence collected but not enough
Frame 3-4: "watching" - strong signals accumulating  
Frame 5+:  "confirming" - multi-signal confirmation achieved
Frame 5+ with scene spike: "alert" - confirmed accident

Key change: require 3+ frames instead of 2 for initial alert
This eliminates transient false positives (car passing) which only last 1-2 frames
```
